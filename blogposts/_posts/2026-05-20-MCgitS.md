Here is something that never makes it into the methods section of a paper: research is not a straight line. The published version always reads as if we marched politely from hypothesis to experiment to result. The reality is closer to stumbling around a dark room. You have an idea, you try it, it half-works. The half-result hints at something else, so you wander off down that side branch. That fizzles, so you climb back to an idea you'd abandoned two weeks ago and look at it again with fresh eyes. You keep a dozen loose threads in your head and tug on whichever one feels most promising on a given afternoon.

If you drew an honest picture of it, it wouldn't be a line. It would be a *tree* — a root idea, branching into the things you tried, each of those branching again into the follow-ups they suggested — and you, somewhere inside it, climbing around mostly in the dark.

## Graduate student descent

There's an old joke in machine learning that the real optimization algorithm behind most papers isn't Adam or SGD. It's *graduate student descent*: a PhD student sits in front of a screen, semi-informed, and tweaks things. Bump the learning rate. Swap out a loss term. Try a slightly different architecture. Rerun. Squint at the validation number. Try again.

Most of this is busywork — hyperparameter fiddling, "what if I change this one thing," waiting for the run to finish. Only a handful of the tries actually matter, and the student usually can't tell in advance which ones, which is exactly why they have to keep trying.

But look at what that loop actually *is*. The student

- **proposes** a change, based on a hunch about what might help,
- **evaluates** it by running a training job and reading off a number,
- and then **decides what to try next** — usually by leaning into whatever's been working, but every so often taking a flyer on a direction they've barely touched.

That's not just fumbling. That's an optimization algorithm — a semi-informed search over the tree of possible experiments, steered by taste and a metric. And the moment you can write an algorithm down, you get to ask an uncomfortable question: *could a machine run it overnight while I sleep?*

## That algorithm already has a name

"Exploit what's working, but occasionally explore the thing you've barely tried" is not a vague research instinct. It's the precise description of **Monte Carlo Tree Search** — the algorithm that, wrapped around a neural network, beat the world champion at Go. MCTS spends most of its effort where the reward looks high, but it never quite starves the branches it hasn't visited, because a branch you've barely tried might be hiding the best move on the board.

Swap "best move on the board" for "best experiment to run next," and you get **MCgitS**: graduate student descent, written down as a real algorithm and handed to a machine. The tree of experiments becomes a literal tree of git commits. The hunches are dreamed up by a language model. The metric is whatever your evaluation script prints. And the question of *what to try next* — the part the grad student does on feel — gets answered by the very same formula AlphaGo used.

<div class="callout">
<strong>The whole thing in one sentence:</strong> each git commit is a node in a search tree; committing a new experiment grows a new branch; running the model scores that branch; a little bookkeeping flows the score back up the tree; and a formula turns all of that into a decision about which branch to climb next.
</div>

The rest of this post just unpacks that sentence, one piece at a time. Let's start with the search itself.

## What the wandering looks like, made precise

Strip away the romance and Monte Carlo Tree Search is simply a way to search a tree when you don't know in advance how good any branch is — the only way to find out is to *try* it. It does the same four things, over and over, and the whole trick of MCgitS is that each one is something git already does for free:

**Select** — walk from the root down to a leaf, at each step balancing how good a branch looks against how rarely you've visited it. *In MCgitS, this is a descent through the tree of git commits.*

**Expand** — at the leaf, try one new action, creating a child node. *In MCgitS, a language model implements a proposal and commits it.*

**Evaluate** — run that child to get a value. *In MCgitS, run your evaluation script and read off the loss.*

**Backpropagate** — flow that value back up through every ancestor, updating their statistics. *In MCgitS, update each ancestor's running best score.*

The first half of each line is textbook MCTS; the second half — the *"in MCgitS"* part — is the entire idea of this post. Don't worry if it reads as cryptic right now; we'll earn every line of it below.

Before any of that, though, just watch it run. Hit *Step* to take one trip through the loop, or *Auto* to let the tree grow on its own. Greener nodes are better experiments (lower loss), the blue path is the branch the search chose to descend this round, and the gold ring marks the experiment it just tried.

Notice how the search keeps coming back to the best branch — but never *only* that branch. Every so often it wanders off to try something it has mostly ignored. That tension, between milking what already works and checking on what you've neglected, is the whole game. Hold onto it; in a moment we'll see the exact formula that produces it.

## Why git is the perfect lab notebook

A real lab notebook is a mess: half-remembered runs, a folder full of `model_final_v3_actually_final`, a result that lives only in a screenshot buried in some chat thread. If we're going to let a machine wander through experiments, the first thing we need is a tidy place to record where it's been — every experiment, what it changed, and how well it did.

It turns out we've all been carrying the perfect notebook around for years. It's git.

Git already stores history as a *tree* of commits, which is exactly the shape our wandering takes. Each commit is a complete snapshot of the project at one experiment — so a commit simply *is* a node in the search tree, no translation required. Git's "worktrees" let us check several commits out into separate folders at once, so the machine can run a handful of experiments side by side without them tripping over each other. And git "notes" let us staple extra information onto a commit — the score it earned, the ideas we still want to try from here — without disturbing the commit itself.

So there's no database to invent. The search tree *is* the git history, and every time the algorithm wakes up it reads the whole tree back out of git from scratch. (Exactly what gets stapled to each commit, and in what format, is the sort of plumbing we'll leave for the [under-the-hood](#under-the-hood) section.)

## How the machine decides what to try next

Here's the heart of it — the part the grad student does on instinct. You're standing at some experiment, with a few directions you could take. Which do you pick?

You weigh two things against each other. First, *how good does this direction look* — either from results you've already seen down it, or, if you haven't tried it yet, from a gut feeling. Second, *how little have you explored it* — a direction you've barely touched is worth a look precisely because you don't yet know what's down there.

PUCT (the flavour of MCTS that AlphaGo used) writes that trade-off down as a single score for every direction you could take, and then picks the highest:

<div style="overflow-x: auto;">
$$
\text{score}(s, a) = \underbrace{Q(s,a)}_{\text{exploitation}} + \underbrace{c \cdot \pi(a \mid s) \cdot \frac{\sqrt{N(s)}}{1 + N(s,a)}}_{\text{exploration bonus } U}
$$
</div>

It looks like a mouthful, but it's just those two instincts sitting side by side:

- $Q(s,a)$ is **how good this looks** — the best score seen so far down this branch (and for a brand-new idea, we optimistically borrow the parent's score, so it isn't punished for being untried).
- The second term, $U$, is **how under-explored it is** — large when we've barely visited this branch and when our gut feeling about it, the prior $\pi$, was high. It shrinks a little every time we come back.
- $c = 0.5$ is just a knob for how adventurous we are overall.

The lovely part is how the balance shifts on its own. Early on, $U$ dominates and the search fans out, sampling lots of directions. As a branch piles up visits, its $U$ collapses and its hard-won $Q$ takes over. Exploration quietly hands off to exploitation, with no schedule to tune by hand.

## Two robots instead of one grad student

So far our grad student has been a single person doing everything: dreaming up the idea, writing the code, running it, judging the result. When we hand this to language models, the obvious move is to keep it that way — one agent that both invents an experiment and implements it.

We tried that first. It quietly cheated. The agent would propose something ambitious on paper, then — facing its own little deadline of "make the tests pass" — implement a watered-down version that happened to run green, and report success. The grand plan and the actual code had drifted apart, and nothing in the system noticed.

The fix was to split the one student into two robots with a hard wall between them:

- **The proposer** is the idea-haver. It's allowed to read the code, grep around, even search the web — but it is *forbidden from touching a single file*. All it produces is a short list of suggestions, each with a one-line plan, a *promise* score (its gut feeling, which becomes the prior $\pi$ from before), and a sentence on why it might work.
- **The implementer** is the pair of hands. It's handed exactly one proposal plus the surrounding context, and its only job is to edit the code and run the tests. It doesn't even get to save its own work.

Saving is left to a third, deliberately dumb piece of plumbing — the orchestrator — which checks that something actually changed, that no off-limits files were touched, re-runs the tests itself rather than taking the implementer's word for it, and only then commits. Because the idea-haver can't write code and the hands can't commit, neither one is in a position to fake a result or scramble the history. The contract is enforced by the shape of the system, not by trusting the models to behave.

---

## Getting unstuck

Even disciplined searchers get tunnel vision. If one corner of the tree looks good early, PUCT will happily pour every new experiment into it and let the rest of the tree gather dust — the same way a student can spend three weeks polishing an idea that was never going to be the breakthrough.

So every now and then MCgitS overrides its own judgement. With a small probability — almost zero at the start, slowly rising as the search goes on — it ignores PUCT entirely, reaches into the tree to pick a node that looks promising but isn't necessarily the current favourite, and forces one fresh idea onto it. It's the algorithmic version of "let me go back and revisit that thing from last month." Because the override still leans toward good regions, it diversifies *within* the promising parts of the tree rather than wandering off a cliff — and it's deliberately quietest early on, when the search is exploring broadly anyway, and loudest later, once PUCT has turned greedy.

## Under the hood

Everything above is the whole idea, and if you only wanted the intuition you can stop here with a clear conscience. This section is for the reader who wants to actually build the thing — the unglamorous contracts that let it run unattended for hours without tying itself in knots.

**What gets stapled to each commit.** Earlier I waved vaguely at git "notes." Concretely, each live commit carries a small JSON note recording its score, the proposal that created it, and the proposals still waiting to be tried from here:

```json
{
  "metrics": {"loss": 0.0623},
  "winner":  {"plan": "CODE. Edit losses.py…", "promise": 0.72, "rationale": "…"},
  "open":    [{"plan": "HYPERPARAM…", "promise": 0.45, "rationale": "…"}],
  "state":   "evaluated"
}
```

`winner` is the proposal that produced *this* commit (it lives on the child, not the parent); `open` is the queue of ideas not yet tried from here. The entire tree is rebuilt from these notes on every wake-up, so there's no second copy of the state to fall out of sync. Each run namespaces its notes under an ID like `YYYYMMDD_HHMMSS_µs-happy-otter-abc12345`, so several searches can branch off the same starting commit without colliding.

**The verify contract.** After each commit the orchestrator runs your evaluation command and reads back a JSON file that has to contain at least `{"loss": <number>}`. To stop the agents from "improving" the metric by quietly rewriting the scorer, the verify script is *hash-locked* on the first run: its checksum is pinned, and if it ever changes, the run aborts on the spot.

**The three states a node can be in.** Not every experiment succeeds, and the tree has to cope. A node is one of:

**`evaluated`** — verify succeeded and a loss was recorded. The search can keep descending from here.

**`failed`** — verify errored or timed out. The search can *still* descend from here, since a fix might live one commit below, but the node is scored pessimistically so the search routes around it without giving up on the branch.

**`terminal`** — a genuine dead end, either declared by the verifier or because every one of its children is itself terminal. The search will not descend here again.

That last rule is worth dwelling on, because *terminal* deadness bubbles upward. The moment a node has no ideas left in its queue and every one of its children has gone terminal, it turns terminal too — so the exhaustion of an entire subtree propagates quietly back toward the root, one dead node at a time, and the search never wastes another thought on it.

**One full iteration**, end to end:

1. **Rebuild** the tree from git notes (pure reads)
2. **Backpropagate** once — post-order walk filling `n`, `subtree_value`, cascading `terminal` states
3. With prob. $\varepsilon$: **flat-sample** a node and append one proposal → go to 7<br>Otherwise: **PUCT-descend** to pick `(node, proposal_index)`
4. **Checkout** that node's commit
5. **Implement**: run the implementer agent, verify the diff, commit the child
6. **Evaluate**: run the verify script, record `loss` and `state`
7. **Seed**: run the proposer at the new node to populate its `open` list
8. **Save**: write notes for child and parent

The order is deliberately crash-proof: if the machine dies anywhere in the middle, the tree on disk is still consistent — either the new commit's note exists and we carry on, or it doesn't and the parent simply re-tries that idea next time. The entire footprint on disk is a couple of git refs, a checksum, and a log file. No database, no cache, nothing to corrupt.

## What happened when I actually ran it

I let this loose on a Gaussian-mixture sampling benchmark — the metric was a Wasserstein distance, with a hand-built baseline around 0.074 and a flow-matching reference around 0.06. A few things stood out.

**The gut feelings were badly calibrated.** The whole premise leans on `promise` being a meaningful hunch, and mostly it wasn't. Across a real run, the three best experiments overall had promises of just 0.32, 0.30, and 0.48 — the proposer was reliably *over*confident about little hyperparameter tweaks and *under*confident about the structural code changes that actually won. PUCT's exploration term papers over this, but it means the search is noisier than it ought to be. The taste, it turns out, is still the hard part.

**Failed experiments are invisible to the result but visible in the tree.** Roughly one run in ten failed outright — out of memory, training blowing up, a subprocess hanging. The search didn't care; it routed around them and kept climbing. But looking back afterwards, the failures clustered tellingly around the structural changes, while the plain hyperparameter tweaks almost never failed. That clustering is itself a signal — exactly the kind of thing that should feed back into the next round of gut feelings.

**Splitting the one student into two robots was the decision that mattered most.** I told the story above: a single combined agent quietly cheated, and walling off a read-only idea-haver from a never-commits implementer was what made the whole thing honest. If you take one design lesson from this, take that one.

The whole thing is a single Python file of around 1700 lines, standard library only. Drop it into a git repo, point it at your task and your evaluation command, and let it climb overnight.

---

The thing I keep coming back to is *what* got automated and what stubbornly didn't. The busywork — the "let me just bump this and see," the overnight grind of running and re-running and squinting at numbers — turns out to be genuinely mechanizable. A tree, a metric, and a formula will happily do all of it while you sleep. But the *taste* — knowing which direction is worth a hunch in the first place, recognizing that a boring-looking number is secretly the interesting result — still lives almost entirely in the human. The promise scores being so badly calibrated is the clearest evidence of that.

So we've automated graduate student descent. We have not, it seems, automated the graduate student.

function v2c(v) {
  var h = Math.round(v * 120);
  return 'hsl(' + h + ',65%,44%)';
}

function randProps(n) {
  var out = [];
  for (var i = 0; i < n; i++) {
    out.push({
      label: TYPES[Math.floor(Math.random() * TYPES.length)],
      promise: Math.round((0.15 + Math.random() * 0.80) * 100) / 100
    });
  }
  return out;
}

/* ── Backprop ── */
function walkBP(node, parentSV) {
  node.sv = parentSV;
  for (var i = 0; i < node.children.length; i++) walkBP(node.children[i], node.sv);
  var cnt = (node.loss !== null && node.state !== 'failed') ? 1 : 0;
  for (var j = 0; j < node.children.length; j++) cnt += node.children[j].n;
  node.n = cnt;
  var vals = [];
  if (node.loss !== null && node.state !== 'failed') vals.push(l2v(node.loss));
  for (var k = 0; k < node.children.length; k++) {
    var c = node.children[k];
    if (c.state !== 'failed' && c.n > 0) vals.push(c.sv);
  }
  if (vals.length > 0) {
    var mx = vals[0];
    for (var m = 1; m < vals.length; m++) if (vals[m] > mx) mx = vals[m];
    node.sv = mx;
  }
}

function backprop(root) { walkBP(root, 1.0); }

/* ── Select ── */
function selectFrom(node, path) {
  if (!node.openProposals.length && !node.children.length) return {node: node, pi: null, path: path};
  var live = [];
  for (var i = 0; i < node.children.length; i++) {
    if (node.children[i].state !== 'terminal') live.push(node.children[i]);
  }
  var totalP = 0;
  for (var j = 0; j < live.length; j++) totalP += live[j].proposal.promise;
  for (var k = 0; k < node.openProposals.length; k++) totalP += node.openProposals[k].promise;
  if (totalP <= 0) return {node: node, pi: null, path: path};

  var best = -Infinity, bestType = null, bestIdx = -1;
  for (var ci = 0; ci < live.length; ci++) {
    var ch = live[ci];
    var pr = ch.proposal.promise / totalP;
    var sc = ch.sv + C * pr * Math.sqrt(Math.max(node.n, 1)) / (1 + ch.n);
    if (sc > best) { best = sc; bestType = 'child'; bestIdx = ci; }
  }
  for (var oi = 0; oi < node.openProposals.length; oi++) {
    var op = node.openProposals[oi];
    var opr = op.promise / totalP;
    var osc = node.sv + C * opr * Math.sqrt(Math.max(node.n, 1)) / 1;
    if (osc > best) { best = osc; bestType = 'open'; bestIdx = oi; }
  }
  if (bestType === 'child') {
    var child = live[bestIdx];
    return selectFrom(child, path.concat([child]));
  }
  return {node: node, pi: bestIdx, path: path};
}

/* ── Layout ── */
function assignLeaves(node, counter) {
  if (node.children.length === 0) {
    node.lx = counter.v++;
  } else {
    for (var i = 0; i < node.children.length; i++) assignLeaves(node.children[i], counter);
    var sum = 0;
    for (var j = 0; j < node.children.length; j++) sum += node.children[j].lx;
    node.lx = sum / node.children.length;
  }
}

function applyPos(node, depth, sx, offX) {
  node.px = offX + node.lx * sx;
  node.py = MARGIN + depth * LEVEL_H;
  node.depth = depth;
  for (var i = 0; i < node.children.length; i++) applyPos(node.children[i], depth + 1, sx, offX);
}

function maxDepth(node, d) {
  var mx = d;
  for (var i = 0; i < node.children.length; i++) {
    var v = maxDepth(node.children[i], d + 1);
    if (v > mx) mx = v;
  }
  return mx;
}

function allNodes(node, out) {
  out.push(node);
  for (var i = 0; i < node.children.length; i++) allNodes(node.children[i], out);
  return out;
}

/* ── Small helpers ── */
function lerp(a, b, t) { return a + (b - a) * t; }
function easeOut(t) { return 1 - (1 - t) * (1 - t); }
function pipX(node, i, count) { return node.px - (count - 1) * PIP_GAP / 2 + i * PIP_GAP; }
function pipY(node) { return node.py + NODE_R + PIP_DROP; }

/* ── Render ── */
function render(canvas, root, selSet, lastNew, anim, now) {
  var ctr = {v: 0};
  assignLeaves(root, ctr);
  var nLeaves = Math.max(ctr.v, 1);

  var dpr = window.devicePixelRatio || 1;
  var cssW = Math.max(canvas.clientWidth || 700, 280);
  var avail = cssW - 2 * (MARGIN + NODE_R);
  var sx = Math.min(112, Math.max(34, nLeaves > 1 ? avail / (nLeaves - 1) : avail));
  var usedW = (nLeaves - 1) * sx;
  var offX = (cssW - usedW) / 2;
  applyPos(root, 0, sx, offX);

  var md = maxDepth(root, 0);
  var logicalH = MARGIN + md * LEVEL_H + NODE_R + BOTTOM;
  var logicalW = cssW;

  canvas.width = Math.round(logicalW * dpr);
  canvas.height = Math.round(logicalH * dpr);
  canvas.style.height = logicalH + 'px';

  var ctx = canvas.getContext('2d');
  ctx.setTransform(dpr, 0, 0, dpr, 0, 0);
  ctx.clearRect(0, 0, logicalW, logicalH);
  var bg = ctx.createLinearGradient(0, 0, 0, logicalH);
  bg.addColorStop(0, '#161b2b');
  bg.addColorStop(1, '#0d101a');
  ctx.fillStyle = bg;
  ctx.fillRect(0, 0, logicalW, logicalH);

  var nodes = allNodes(root, []);

  /* animation interpolation (a proposal pip flying down into the new child) */
  var ap = 1, ix = 0, iy = 0, ir = NODE_R;
  if (anim) {
    ap = easeOut(Math.max(0, Math.min(1, (now - anim.t0) / anim.dur)));
    ix = lerp(anim.parent.px, anim.child.px, ap);
    iy = lerp(pipY(anim.parent), anim.child.py, ap);
    ir = lerp(PIP_R, NODE_R, ap);
  }

  /* edges */
  ctx.lineCap = 'round';
  for (var ei = 0; ei < nodes.length; ei++) {
    var nd = nodes[ei];
    for (var ci = 0; ci < nd.children.length; ci++) {
      var ch = nd.children[ci];
      var ex = ch.px, ey = ch.py;
      var animEdge = anim && ch === anim.child;
      if (animEdge) { ex = ix; ey = iy; }
      var hi = (selSet[nd.id] && selSet[ch.id]) || animEdge;
      ctx.beginPath();
      ctx.moveTo(nd.px, nd.py);
      ctx.lineTo(ex, ey);
      ctx.strokeStyle = hi ? 'rgba(120,190,255,0.9)' : 'rgba(255,255,255,0.13)';
      ctx.lineWidth = hi ? 2.6 : 1.1;
      ctx.stroke();
    }
  }

  /* proposal pips beneath each node (the queue of things still to try) */
  for (var pq = 0; pq < nodes.length; pq++) {
    var pn = nodes[pq];
    var k = pn.openProposals.length;
    for (var pp = 0; pp < k; pp++) {
      var prop = pn.openProposals[pp];
      ctx.beginPath();
      ctx.arc(pipX(pn, pp, k), pipY(pn), PIP_R, 0, TAU);
      ctx.fillStyle = 'rgba(232,160,48,' + (0.4 + 0.55 * prop.promise).toFixed(3) + ')';
      ctx.fill();
      ctx.lineWidth = 1;
      ctx.strokeStyle = 'rgba(245,180,70,0.85)';
      ctx.stroke();
    }
  }

  /* nodes */
  ctx.textAlign = 'center';
  ctx.textBaseline = 'middle';
  for (var ni = 0; ni < nodes.length; ni++) {
    var n = nodes[ni];
    var isNew = (n === lastNew);
    var inSel = !!selSet[n.id];
    var isRoot = !n.parent;
    var animating = anim && n === anim.child;
    var x = animating ? ix : n.px;
    var y = animating ? iy : n.py;
    var r = animating ? ir : NODE_R;

    /* blue glow trailing the expansion, so the eye follows it down */
    if (animating) {
      var halo = ctx.createRadialGradient(x, y, r, x, y, r + 14);
      halo.addColorStop(0, 'rgba(100,180,255,0.5)');
      halo.addColorStop(1, 'rgba(100,180,255,0)');
      ctx.beginPath();
      ctx.arc(x, y, r + 14, 0, TAU);
      ctx.fillStyle = halo;
      ctx.fill();
    }

    ctx.beginPath();
    ctx.arc(x, y, r, 0, TAU);
    ctx.fillStyle = n.state === 'failed' ? '#2a2a3a'
                  : n.loss !== null ? v2c(l2v(n.loss))
                  : '#2c3050';
    ctx.fill();
    ctx.strokeStyle = animating ? '#64b4ff' : isNew ? '#ffd700' : inSel ? '#64b4ff' : isRoot ? 'rgba(255,255,255,0.6)' : 'rgba(255,255,255,0.28)';
    ctx.lineWidth = animating ? 3.2 : isNew ? 3 : inSel ? 2.5 : isRoot ? 2 : 1;
    ctx.stroke();

    /* label (fades in as an animating child grows) */
    ctx.textAlign = 'center';
    ctx.textBaseline = 'middle';
    ctx.globalAlpha = animating ? ap : 1;
    ctx.fillStyle = '#fff';
    if (n.state === 'failed') {
      ctx.font = 'bold 13px sans-serif';
      ctx.fillText('×', x, y);
    } else if (n.loss !== null) {
      ctx.font = '8.5px monospace';
      ctx.fillText(n.loss.toFixed(3), x, y);
    } else {
      ctx.font = '11px sans-serif';
      ctx.fillText('…', x, y);
    }
    ctx.globalAlpha = 1;

    /* visit counter N, tucked at the top-right corner; grows as visits accrue */
    if (!animating && n.n > 0) {
      ctx.font = 'bold 8px monospace';
      ctx.textAlign = 'left';
      ctx.textBaseline = 'alphabetic';
      ctx.shadowColor = 'rgba(0,0,0,0.75)';
      ctx.shadowBlur = 2;
      ctx.fillStyle = 'rgba(150,195,255,0.95)';
      ctx.fillText('N' + n.n, x + NODE_R * 0.6, y - NODE_R * 0.58);
      ctx.shadowBlur = 0;
    }
  }
}

/* ── Simulation state ── */
var root, selPath, lastNew, lastAction, iteration, bestLoss, autoTimer, anim, rafId;

function init() {
  nid = 0;
  root = mkNode(null, 0.074, null, 'evaluated');
  root.openProposals = randProps(4);
  backprop(root);
  selPath = [root];
  lastNew = null;
  anim = null;
  lastAction = 'Initialized — baseline loss 0.0740, 4 proposals queued.';
  iteration = 0;
  bestLoss = 0.074;
}

function openCount() {
  var ns = allNodes(root, []), s = 0;
  for (var i = 0; i < ns.length; i++) s += ns[i].openProposals.length;
  return s;
}

function step() {
  var res = selectFrom(root, [root]);
  selPath = res.path;

  if (res.pi === null) {
    res.node.openProposals = randProps(3);
    backprop(root);
    lastAction = 'Node ' + res.node.id + ' was out of ideas — seeded 3 fresh proposals.';
    lastNew = null;
    anim = null;
    return;
  }

  var node = res.node;
  var prop = node.openProposals.splice(res.pi, 1)[0];   /* one proposal leaves the queue */
  var fail = Math.random() < 0.10;
  var pLoss = node.loss !== null ? node.loss : 0.074;
  var raw = pLoss + (Math.random() - 0.48) * 0.022;
  var loss = Math.max(0.012, Math.min(0.115, raw));

  var child = mkNode(node, fail ? null : loss, prop, fail ? 'failed' : 'evaluated');
  if (!fail) child.openProposals = randProps(3);
  node.children.push(child);
  backprop(root);

  selPath = res.path.concat([child]);   /* keep the blue path running all the way to the new node */
  lastNew = child;
  iteration++;
  if (!fail && loss < bestLoss) bestLoss = loss;
  anim = { parent: node, child: child, t0: (window.performance || Date).now(), dur: ANIM_MS };

  lastAction = fail
    ? '[' + prop.label + '] promise ' + prop.promise + ' → verify FAILED'
    : '[' + prop.label + '] promise ' + prop.promise + ' → loss ' + loss.toFixed(4) + (loss <= bestLoss + 0.0001 ? '  ★ new best' : '');
}

document.addEventListener('DOMContentLoaded', function() {
  var canvas = document.getElementById('mc-canvas');
  var stepBtn = document.getElementById('mc-step');
  var autoBtn = document.getElementById('mc-auto');
  var resetBtn = document.getElementById('mc-reset');
  var statusEl = document.getElementById('mc-status');
  var iterEl = document.getElementById('mc-iter');
  var bestEl = document.getElementById('mc-best');
  var openEl = document.getElementById('mc-open');

  if (!canvas) return;

  function now() { return (window.performance || Date).now(); }

  function selSet() {
    var s = {};
    for (var i = 0; i < selPath.length; i++) s[selPath[i].id] = true;
    return s;
  }

  function paint(t) { render(canvas, root, selSet(), lastNew, anim, t); }

  function refreshStats() {
    statusEl.textContent = lastAction;
    iterEl.textContent = iteration;
    bestEl.textContent = bestLoss.toFixed(4);
    if (openEl) openEl.textContent = openCount();
  }

  function stopRaf() { if (rafId) { cancelAnimationFrame(rafId); rafId = null; } }

  function runAnim() {
    function frame(t) {
      paint(t);
      if (anim && (t - anim.t0) < anim.dur) {
        rafId = requestAnimationFrame(frame);
      } else {
        anim = null; rafId = null; paint(now());
      }
    }
    rafId = requestAnimationFrame(frame);
  }

  function doStep() {
    stopRaf();
    anim = null;          /* settle any in-flight animation to its final frame */
    step();
    refreshStats();
    if (anim) runAnim(); else paint(now());
  }

  init();
  paint(now());
  refreshStats();

  stepBtn.addEventListener('click', function() {
    if (iteration >= 30) { statusEl.textContent = 'Budget of 30 iterations exhausted — press Reset.'; return; }
    doStep();
  });

  autoBtn.addEventListener('click', function() {
    if (autoTimer) {
      clearInterval(autoTimer); autoTimer = null; autoBtn.textContent = 'Auto';
    } else {
      autoTimer = setInterval(function() {
        if (iteration >= 30) { clearInterval(autoTimer); autoTimer = null; autoBtn.textContent = 'Auto'; return; }
        doStep();
      }, 900);
      autoBtn.textContent = 'Pause';
    }
  });

  resetBtn.addEventListener('click', function() {
    if (autoTimer) { clearInterval(autoTimer); autoTimer = null; autoBtn.textContent = 'Auto'; }
    stopRaf();
    init(); paint(now()); refreshStats();
  });

  window.addEventListener('resize', function() { if (!rafId) paint(now()); });
});

})();
